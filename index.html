System Overview

JiraAgent is an autonomous AI coding agent that watches Jira for tickets moved to In Progress, then runs a full 4-phase pipeline — Understand → Implement → Validate → Ship — with zero manual steps until the human review gate.

Entry Point
src/orchestrator/index.ts → runPipeline(ticketKey)
Webhook mode
npm run server — listens on port 3000, polls Jira every 30s
Manual mode
npx ts-node src/orchestrator/index.ts TICKET-KEY
LLM Backend
Anthropic Claude via CodeMie proxy → http://127.0.0.1:4001
Target Repo
Configured in repos.config.json keyed by Jira project prefix
Branch naming
feature/TICKET-KEY (or TICKET-KEY-agent in test mode)
Design principle: The agent commits and pushes NOTHING until a human explicitly types "push", "yes", "lgtm", or "approve" at the interactive terminal prompt. All LLM-generated code stays local until approved.

Ticket Intake & Git Setup

Before any agent runs, the orchestrator handles two concerns: getting the ticket and preparing the git workspace.

Webhook / Polling
  • Webhook server on :3000 receives POST from Jira Automation
  • OR polls getAssignedTickets() every 30s (configurable via POLL_INTERVAL_MS)
  • New tickets go into an in-memory queue
  • processQueue() drains queue sequentially — one ticket at a time
  • Deduplication prevents same ticket being processed twice
Git Branch Setup
  • resolveRepo(ticketKey) → looks up local repo path from repos.config.json
  • Checks out develop, runs git pull origin/develop
  • Creates feature/TICKET-KEY (or reuses if exists)
  • Test mode suffix: feature/TICKET-KEY-agent when testSuffix is set
  • Push is always a separate explicit step — never auto-pushed
Phase 1 — Understand
1
UNDERSTAND
Ticket → Structured Requirements → Codebase Context → Implementation Brief
4 Agents

Converts a raw Jira ticket into a complete understanding of what needs to change and where. All agents run sequentially; each feeds the next.

ticketParser → complexityGate → requirementDecomposer → codebaseScanner → contextBrief (medium+)
Phase 2 — Implement
2
IMPLEMENT
Plan → Write Code → Cross-File Audit → Write Tests
4 Agents

Takes the task list from Phase 1 and produces actual file changes on disk. Each file is written sequentially so later files can reference earlier ones.

specWriter (small+) → codeGenerator → crossFileAuditor → testGenerator (small+)
Phase 3 — Validate
3
VALIDATE
Static Analysis → Test Loop (max 3 retries) → Conflict Resolution → Code Review
4 Agents + Retry Loop

First pass: full validation with ESLint, Jest, and AI review. Subsequent passes (after human revision): fast re-validation — consistency + static + review only (no Jest re-run).

First Pass (full)
  1. staticAnalyzer → ESLint + AI review
  2. runTests (Jest) → if fail: autoFixer × each suite → retry (max 3)
  3. conflictResolver → only if merge conflicts detected
  4. codeReviewer → AC coverage + security + score/10
Post-Revision (fast)
  1. crossFileAuditor → consistency check
  2. staticAnalyzer → static review only
  3. codeReviewer → final quality gate
  4. (Jest skipped — assumes feedback was targeted)
Phase 4 — Human Review Gate
4
HUMAN REVIEW GATE
Interactive terminal prompt — developer approves, rejects, or gives feedback
Human Decision

The only blocking step. Prints a diff summary and Phase 3 scores, then waits for developer input. The input itself is classified by keyword matching.

What you type Action Result
push / yes / lgtm / approve Approve commit + push to target branch
skip / s / local Skip-push commit locally, no push
no / reject / abort Reject discard all changes, exit
anything else Revise input becomes feedback → revisionAgent → loop back to Phase 3
Max 5 revision cycles. After cycle 5 the agent enters a final review loop without further revision capability.
Complexity Gate
assessComplexity() — Controls Pipeline Behavior

Scored from ticket fields after ticketParser runs. Determines model tier, token budgets, task limits, and which pipeline steps to skip.

Level AC Count Change Signals Story Points maxTasks maxTokens maxFiles Synthesis Planning Tests
trivial ≤1 ≤1 ≤1 2 4,096 3 skip skip skip
small ≤2 ≤3 ≤3 5 8,192 8 skip ✓ minimal
medium ≤5 ≤6 ≤5 12 16,384 15 ✓ ✓ full
large else else else 20 32,768 20 ✓ ✓ full

Change signals = uiChanges.length + apiChanges.length + dataChanges.length. New component/API creation auto-upgrades to medium+.

Understand Agents — Detail
ticketParser
Haiku Sonnet
Purpose
Extract only what is explicitly stated in the Jira ticket. No inference, no expansion.
Input
JiraTicket — key, summary, description, acceptanceCriteria, comments[], links[]
Output
ParsedRequirements — userStory, acceptanceCriteria[], technicalNotes[], uiChanges[], apiChanges[], dataChanges[], outOfScope[]
Key rule
System prompt forbids inference. "change button text" → uiChanges=["change button text"], all other arrays empty.
Edge case
Regex JSON extraction with try/catch. Returns AgentResult<ParsedRequirements> — pipeline exits on failure.
complexityGate
No LLM
Purpose
Score ticket complexity to control model selection, token budgets, task limits, and which steps to skip.
Input
JiraTicket + ParsedRequirements (acCount, changeSignals, storyPoints)
Output
ComplexityConfig — level, maxTasks, maxTokens, maxCodebaseFiles, skipSynthesis, skipSeparatePlanning, testGeneration
Logic
Pure function — no LLM. AC count + change signals + story points → trivial/small/medium/large. New component/API creation → auto-upgrades to medium+.
requirementDecomposer
HaikuSonnetOpus
Purpose
Convert requirements into the minimum set of file-level tasks. Prefers fewer, targeted tasks over many small ones.
Input
ParsedRequirements + flat repo file structure (all paths) + maxTasks
Output
ImplementationTask[] — each with id, description, filePath (must exist in repo), changeType (modify/create/delete), dependsOn[]
Key rule
One task per file. Ordered by dependency. Skips test tasks for trivial changes. Trims if over maxTasks with a warning.
codebaseScanner
HaikuSonnet
Purpose
Identify which files matter for this feature, read them, understand architecture and tech stack.
Input
ParsedRequirements + repoPath + maxFiles (from complexity)
Output
CodebaseContext — relevantFiles[], dependencyMap, architectureNotes, techStack[], testingFramework
2-step LLM
Step 1: identify relevant files from flat list. Step 2: describe files in plain text → convert to JSON (avoids code-in-JSON escaping bugs).
Enrichment
Reads actual package.json to verify tech stack (React, Redux, Jest, etc.) — overrides LLM guesses with real deps.
Fallback
If JSON parse fails → builds minimal CodebaseContext from file list with 'Identified as relevant' reason.
contextBrief
HaikuSonnetmedium+ only
Purpose
Synthesize all Phase 1 outputs into a human-readable implementation brief. Skipped for trivial/small.
Input
ticket + requirements + codebaseContext + tasks[]
Output
context_output.md written to cwd: brief, keyDecisions[], implementationOrder[], risks[], contextSummary
Edge case
Falls back to raw LLM text if JSON parse fails — still writes the file.
Implement Agents — Detail
specWriter
HaikuSonnetOpussmall+ only
Purpose
Write per-file implementation specs BEFORE code generation. Skipped for trivial (inline specs used instead).
Input
ImplementationTask + requirements + codebaseContext + current file content (first 8000 chars) + dependent file contents (first 4000 chars each)
Output
Record<taskId, specString> — JSON per file: imports, exports, implementation notes, style notes
Key rule
Sequential per task. Each spec reads dependencies already specified. "Only cover logic the AC explicitly requires."
codeGenerator ⭐ Most Critical
HaikuSonnetOpus
Purpose
Write complete file content for each task. Directly produces all code changes — the most impactful agent in the pipeline.
Input
task + spec + requirements + codebaseContext + previously-written files (dependency awareness) + 2 style reference files
Output
CodeChange[] — each with filePath, originalContent, newContent, diff, explanation. Written to disk immediately.
Sequential
Files written one at a time so each file can reference previously written files in the same run.
Style
Reads 2 existing "modify" files as style references. Strips markdown fences from output with regex cleanup.
crossFileAuditor
HaikuSonnet
Purpose
Catch cross-file bugs that single-file generation misses — import/export mismatches, prop chains, Redux shape, wrong names.
Input
All CodeChange[] files, read fresh from disk
Output
Fixed files written in-place to disk. Up to 6 files patched per run.
2-step
Step 1: plain-text audit (avoids JSON+code escaping issues). Step 2: apply fix per affected file. If "NO ISSUES FOUND" → early exit.
Checks
Constants not exported, Redux initialState vs selector shape, parent→child prop mismatches, wrong import paths, function name mismatches
testGenerator
HaikuSonnetsmall+ only
Purpose
Write focused unit tests that verify acceptance criteria. Skipped for trivial. "Test what changed, not the entire component."
Input
test ImplementationTask + corresponding impl CodeChange + requirements + codebaseContext
Output
Test file CodeChange[] written to disk
Pattern
Reads existing test file from repo if found — follows its exact mock patterns. Defaults: jest.mock, CSS module mocks, redux-mock-store, React Testing Library.
Coverage
Happy path + 1 error case per AC. No edge cases unless AC explicitly names them.
Validate Agents — Detail
staticAnalyzer
HaikuSonnet
Purpose
Dual-mode static analysis: real ESLint + AI review for things ESLint can't catch.
Input
CodeChange[] + repoPath
Output
DebuggingResult — issues: LintIssue[], passed: boolean, summary string
ESLint
execSync with 30s timeout, JSON output format, max-warnings 50. Parses severity 2 = error, 1 = warning.
AI review
Checks: missing imports, undefined vars, React hook misuse (conditional, missing deps), Redux action type mismatches, wrong prop shapes, async/await errors.
autoFixer
HaikuSonnetOpus
Purpose
Fix a specific failing file given test failures + lint issues + review comments.
Input
filePath + failures (TestFailure[]) + lintIssues (LintIssue[]) + reviewComments (ReviewComment[])
Output
Fixed CodeChange written to disk
Logic
Builds unified issue list from all 3 sources, filtered to this file. If no issues → no-op return. Called per failed test suite in the 3-retry loop.
Missing module
Special case: if error contains "Cannot find module", resolves the missing module path rather than the test file path.
conflictResolver
HaikuSonnetonly if conflicts
Purpose
Resolve git merge conflicts. Only invoked if getMergeConflicts() returns > 0 conflicts.
Input
repoPath — reads files with conflict markers directly
Output
Resolved files written to disk
Model note
Only gets Sonnet on large tickets — Haiku for trivial/small/medium. Conflict resolution is rarely needed and relatively mechanical.
codeReviewer
HaikuSonnetOpus
Purpose
Final automated quality gate — AC coverage, security, correctness scoring. Runs every validation cycle.
Input
CodeChange[] + ParsedRequirements
Output
ReviewResult — approved:bool, acCoverage[], comments[], blockers[], suggestions[], overallScore/10
Checks
Every AC has covering code; bugs/logic errors; security (XSS, auth bypass, missing validation); React/Redux best practices; edge cases (empty state, loading, error).
Display
Score + AC fraction shown in console. Blockers listed individually. Score shown to developer in Phase 4 prompt.
humanGate
InteractiveNo LLM
Purpose
Final human decision point. Nothing is committed or pushed until this step approves.
Input
CodeChange[] + phase3ReportPath + featureBranch + revisionCycle count
Output
HumanReviewDecision — action: approve|reject|skip|revise + targetBranch + comments + feedback
Classification
Pure keyword matching. "push"/"yes"/"lgtm" → approve. "no"/"reject" → reject. "skip"/"s" → local commit. Anything else → revise (input = feedback).
UX
Prints file diff table (modified/created per file with line count). Reads phase3_report.md for score + AC coverage display.
revisionAgent
SonnetOpuscodeGenerator model
Purpose
Apply human feedback precisely to the code. No scope creep. "Make ONLY what the feedback asks."
Input
feedback string + CodeChange[] + repoPath
Output
Updated CodeChange[] written to disk
2-step
Step 1 (triage): identify which files the feedback targets → returns [{filePath, instruction}]. Step 2: apply per file.
Fallback
If triage returns empty array → applies feedback to ALL files in changes. Prevents silent no-op on vague feedback.
Data Flow — TypeScript Interfaces

Each phase produces a typed output that flows into the next. All agents return AgentResult<T> — the pipeline exits on success: false.

JiraTicket → ticketParser → ParsedRequirements → requirementDecomposer → ImplementationTask[] → codeGenerator → CodeChange[] → codeReviewer → ReviewResult → humanGate → HumanReviewDecision

  acceptanceCriteria, status
